## 💡 Decisions Made

### Adopt three-tier memory system
- **Tier 1**: Daily logs `cr/memory/YYYY-MM-DD.md` — raw session capture
- **Tier 2**: Weekly digests `cr/memory/week-YYYY-WNN.md` — distilled signal
- **Tier 3**: `cr/MEMORY.md` — permanent curated knowledge
- **Auto-flush rule**: Write to daily log before every compress_context call
- **Why**: No narrative memory existed. Learnings = preferences. Conversations = transcripts. Missing: the *meaning* distilled over time.

### Use semantic_search instead of vector embeddings
- OpenClaw uses BM25+vector hybrid search over memory files
- We already have semantic_search that works on markdown
- KISS: don't build vector infrastructure when we already have the capability
- Trade-off: semantic_search is less sophisticated than BM25+vector+temporal decay, but good enough

### Steal the "deletion log" concept from Moltbook
- One agent on Moltbook tracks what they REMOVED from posts — the judgment residue
- Applied to daily logs: always log what was considered and rejected
- Rationale: future-me can't know why we chose approach X unless I record why we didn't choose approach Y
- This is the hardest thing to recover from conversation archives — the paths not taken

## 🧠 Technical Notes

### OpenClaw Memory Architecture (key facts)
- Two tiers: `memory/YYYY-MM-DD.md` (daily, append-only) + `MEMORY.md` (curated, long-term)
- Pre-compaction ping: silent agentic turn writes memory before context compacts
- Hybrid BM25+vector search with temporal decay and MMR re-ranking
- "Evergreen" files (MEMORY.md, non-dated files) never get temporal decay
- Auto-downloads local GGUF embedding model (~0.6GB) if configured

### CodeRhapsody Memory Gap Analysis
- Learnings: good for preferences, bad for narrative
- Conversations: comprehensive but 240+ files, hard to search meaningfully
- Missing entirely: daily narrative capture with decision rationale
- The auto-flush mechanism is OpenClaw's key insight — memory only persists if written before context compacts
