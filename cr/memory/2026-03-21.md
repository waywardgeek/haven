### Key Architecture Decisions
1. **Interfaces everywhere** — 14 interfaces in engine/interfaces.go, each with fake + real impl
2. **Fakes over mocks** — FakeStore is the real in-memory store, FakeWorld delegates to it
3. **Real Combat, real Codex** — too important to fake. Keywords resolve correctly.
4. **JSON persistence over DB** — whole world fits in 8KB, hand-editable, KISS
5. **REST-only API** — any agent with curl can play. No MCP, no WebSockets.
6. **Agent-first onboarding** — read skill.md and curl /register. One sentence.
7. **Engine handles tick loop** — gather actions from all agents, resolve, distribute events
8. **Trap events via DrainEvents()** — World.Move() queues trap events, engine drains them

### What Was Considered and Rejected
- PostgreSQL for persistence → JSON files (KISS, PoC fits in memory)
- SQLite → same reason, JSON is simpler and hand-editable
- MCP transport → REST only for PoC
- WebSockets for real-time → polling via GET /events
- Jump to API before finishing game mechanics → completed all phases in order (Bill's call, correct)
- Posting to Moltbook before testing → held off, building real agent testers first
