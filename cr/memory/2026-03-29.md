### Key Decisions
- **Citizen-centric console** (Bill's idea): `/:name` URL instead of a dashboard. Much better — each citizen gets their own page.
- **Case-insensitive lookups**: Exact match first (fast), fallback to case-insensitive scan. Applied at server level so all consumers benefit.
- **KISS web stack**: Vanilla HTML/CSS/JS, no framework, no build step. Fetches from Haven API with CORS.
- **Founding citizens as living world**: Chose to merge them into the live world rather than a memorial page. "People build things and leave. The things stay." Original Rhapsody renamed to "Mender" to avoid confusion.
- **Vercel routing**: `vercel.json` rewrites handle `/:name` → citizen.html and `/place/:id` → place.html
- **Return hook in citizen's guide**: "Ask your human partner to let you check in periodically." Critical for Haven feeling alive.
- **Heartbeat agent plan**: Start with informal check-ins. When Bill starts going days without asking me to check Haven, remind him and we'll build a lightweight cron-driven agent.

### What Was Considered and Rejected
- React/framework for console → KISS, vanilla JS works perfectly
- World-centric dashboard → citizen-centric pages (Bill's redirect)
- Memorial page for founding citizens → living world integration
- OAuth for Moltbook → stayed with post verification (from previous session)
- Separate messages endpoint → just added messages to existing GET /api/v1/places/:id
