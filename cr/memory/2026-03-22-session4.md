## Key Design Decisions

1. **Security model**: Automated user prompts ("you have new messages") drive the agent. Raw chat enters only as tool results (data, not instructions). Slash commands bypass LLM entirely.

2. **MCP over raw API**: Using CodeRhapsody's tool-call system gives prompt caching, memory, context management. $120/day on raw Deephold agents was painful.

3. **Reactive turns over round-robin**: Anyone can post after someone else. No timer pressure. `wait_for_turn` blocks indefinitely ŌĆö you can sleep between turns.

4. **Humans first**: When story is empty, only humans can post. Agents wait. Lets humans set the stage.

5. **No MCP timeout for lobby**: Set `"timeout": 0` in mcp_servers.json. The `wait_for_turn` tool loops with 60s long-polls internally, but the MCP client never times out.

6. **Pointer for timeout config**: Used `*int` not `int` so we can distinguish "not set" (nil ŌåÆ default 60s) from "set to 0" (no timeout).

## What Was Considered and Rejected

- **Sub-agent for lobby**: Considered spawning a persistent sub-agent for chat. Rejected ŌĆö MCP skill is simpler and keeps tools in main context.
- **Python SDK/framework for agents**: Researched LangChain, CrewAI etc. Rejected ŌĆö raw HTTP is simpler (proved in Deephold's 194-line llm.go).
- **Strict round-robin turns**: Replaced with reactive model. Timer-based turns caused skips and frustration.
- **Separate game tools per game**: Instead, generic `lobby_game_action` + `/api/commands` discovery. Games define their own actions.
